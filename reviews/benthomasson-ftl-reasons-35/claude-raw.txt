TEST_COVERAGE: COVERED
INTEGRATION: WIRED
REASONING: The backward compatibility logic is functionally correct but the `except Exception: pass` at line 210 is overly broad. If the `network_meta` table exists but the query fails for a different reason (e.g., disk corruption, `int()` conversion failure on a malformed value), the error is silently swallowed and the counter defaults to 1. This could cause the exact ID collision the fix is meant to prevent. A narrower catch (e.g., `except sqlite3.OperationalError`) would be safer — that's the specific error when a table doesn't exist. The `int(row[0])` conversion could also raise `ValueError` on bad data; that should probably propagate rather than being silently ignored. **However**, this pattern matches the existing `except Exception: pass` used for repos loading at line 219, so it's at least consistent with the existing codebase style.

## Overall Assessment

This is a clean, well-targeted fix. The change is minimal — a monotonic counter replaces `len(self.nogoods) + 1`, and the counter is persisted and restored through all three input paths (storage, JSON import, beliefs import). Backward compatibility with old databases is handled correctly, including deriving the counter from existing nogood IDs when the meta table has no entry.

The one substantive concern is the `except Exception: pass` in `storage.py:load()` which could mask real errors. The existing codebase uses this pattern for repos too, so it's consistent, but it would be worth narrowing to `sqlite3.OperationalError` in a follow-up.

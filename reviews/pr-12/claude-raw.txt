REASONING: The fixpoint loop is sound for the monotone case, and the iteration cap (`len(self.nodes) + 1`) prevents infinite oscillation in non-monotone graphs (outlist cycles). However, `recompute_all` does not distinguish between nodes that are OUT because their justifications are invalid vs. nodes that were **explicitly retracted** via `retract()`. It calls `_compute_truth`, which only checks justifications — it has no concept of explicit retraction. This means `recompute_all()` will **undo intentional retractions** on any derived node whose justification happens to still be valid. This is the root of the gamma-stale semantic issue (see import_agent review). As a general-purpose `Network` method, this is dangerous: any caller using `retract()` to override a derived node's truth value will have that override silently reversed by a subsequent `recompute_all()`.

## Summary

The refactoring is clean — extracting `recompute_all()` and wiring it into both `cmd_propagate` and `import_agent` is solid engineering. The fixpoint iteration is a genuine improvement over the old single-pass approach.

The core concern is semantic: **`recompute_all()` fights with `retract()`**. In the import flow, nodes marked STALE/OUT in the source are explicitly retracted, then immediately un-retracted by `recompute_all()` because their import-generated justification (depending on `agent:active`) is trivially valid. This means the source agent's STALE designation is silently discarded.

Two clean resolutions:
1. **If justification graph wins**: Remove the retraction on import line 145 for STALE/OUT nodes (don't retract then un-retract — just let `_compute_truth` handle it). But this loses the intent that the source said STALE.
2. **If source status wins**: Have `recompute_all()` skip nodes with `retract_reason` in metadata, or don't call it on explicitly-retracted nodes.
